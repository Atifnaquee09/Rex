1 · System topology

One Node service on a single VPS runs the Slack bot, ticket worker, and dashboard. Embeddings are local; only Claude is remote.

2 · Slack message pipeline

Cheap, deterministic checks run first. The model is only called when a message is genuinely a task, an admin action, or open conversation.

flowchart TD
  A["Slack mention or DM"] --> D{"Duplicate event?"}
  D -->|yes| X["drop"]
  D -->|no| E{"Contains '::' ?"}
  E -->|yes| T1["File ticket directly"]
  E -->|no| TR{"Trivial - hi/thanks?"}
  TR -->|yes| LITE["Light reply - skip roster + KB"]
  TR -->|no| ADM{"Admin request? verb + context"}
  ADM -->|yes| ADMDO["Create/archive/invite/kick - admins only"]
  ADM -->|no| REL{"Mentions another person?"}
  REL -->|yes| RELDO["Relay message to them"]
  REL -->|no| CTX["Gather: thread history + team roster + KB recall"]
  CTX --> TRI{"Triage - model"}
  TRI -->|TASK| T2["File ticket - queue for worker"]
  TRI -->|CHAT| CH["Reply in Rex's persona"]

3 · Ticket execution

"Run with Rex" queues a ticket. The worker claims the highest-priority one and runs the Agent SDK against the repo. Results land in In Review.

flowchart LR
  RUN["Run with Rex / Slack ticket"] --> Q[("Queue - SQLite")]
  WORKER["Worker loop - every 3s"] --> Q
  WORKER --> AGENT["Agent SDK - bypass + guardrails - explorer/reviewer subagents"]
  AGENT --> WS[("Repo workspace")]
  AGENT --> REC[("Run record - tokens, cost, result")]
  AGENT -->|success| IR["In Review"]
  AGENT -->|failure| TODO["To Do"]
  AGENT --> SUP["Slack update with #id"]

4 · Knowledge base (semantic recall)

Embeddings run on-box (free, zero Claude tokens). Recall is selective top-K, so the prompt stays small no matter how big the knowledge base grows.

flowchart LR
  ADD["Add knowledge"] --> EMB1["Local embed 384-d"]
  EMB1 --> STORE[("pgvector store")]
  MSG["Incoming message"] --> EMB2["Local embed query"]
  EMB2 --> SRCH["Top-K cosine search + threshold"]
  SRCH --> STORE
  SRCH --> INJ["Inject top snippets into prompt"]
  INJ --> REPLY["Grounded reply"]
